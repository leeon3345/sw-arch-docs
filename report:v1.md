## 0. 프로젝트 기획 배경: 기존 솔루션의 문제 상황 (Problem Statement)

본 프로젝트는 실제 실무 환경(하이웍스, 네이버웍스 등)에서 겪은 명확한 불편함(Pain Points)에서 출발했습니다. 대기업 맞춤형으로 설계된 기존 그룹웨어가 **중소기업/1인 기업** 환경에서 유발하는 한계점을 4가지 영역으로 구조화했습니다.

| 분류 | 구체적인 문제 상황 (Symptom) | 실무적 영향 및 결과 (Impact) |
| :--- | :--- | :--- |
| **파편화된 UX** | • 기능을 클릭할 때마다 브라우저의 새로운 탭이 무한히 생성됨.<br>• 이전 메신저 히스토리를 거슬러 올라가거나 검색하기 매우 힘든 구조.<br>• 사파리(Safari) 등 특정 브라우저에서의 호환성 문제 발생. | 업무 맥락(Context)이 지속적으로 단절되며, 저사양 PC 사용이 잦은 소규모 기업 환경에서 브라우저 메모리 부하를 유발함. |
| **소통 및 협업 한계** | • 조직도나 사용자 선택 창에서 바로 '그룹 채팅방'을 개설할 수 없음.<br>• 프로필(발송자) 클릭 시 즉시 메일 쓰기나 일정 보기 등의 퀵 액션으로 이어지지 않음.<br>• 세분화된 연락처 검색 필터가 없고 매칭 과정이 불투명함. | 직관적인 소통 경로가 차단되어, 메신저를 통한 '빠른 협업'이라는 본연의 가치를 발휘하지 못함. |
| **외부 연동의 부재** | • 중소기업 실무의 표준인 '구글 캘린더'와의 완벽한 100% 동기화 미지원.<br>• 외부 사용자(Guest)를 묶어서 별도로 그룹화하거나 간편하게 초대하는 기능 부재. | 내부 시스템과 외부 툴을 이중으로 관리해야 하는 리소스 낭비가 발생하며, 외부 클라이언트와의 소통을 저해함. |
| **보안 및 정책 한계** | • **[방어적 설계 부재]** 고객이 DNS 설정 시 CNAME과 MX 레코드를 충돌시켜도 시스템이 경고하지 않아 메일 수신이 영구 차단됨.<br>• **[프라이버시]** 관리자 계정으로 타 직원의 대화 내용을 열람할 수 있는 구조. | 시스템 결함 방치로 인한 비즈니스 기회 상실 위험을 초래하며, 직원들의 프라이버시 침해 우려로 메신저 채택률을 떨어뜨림. |

> **💡 도출 결론 (Project Vision):** 위 문제들을 해결하기 위해, 본 프로젝트는 **① 화면 전환 없는 SPA 기반의 직관적 UX**, **② 외부인과의 유연한 소통을 위한 매직 링크**, **③ 방어적 설계가 적용된 인프라 및 서드파티 연동**을 제공하는 **'소규모 기업 맞춤형 메신저 코어 그룹웨어'**를 설계합니다.

## 산출물 A — 벤치마킹 및 설계 근거 비교표

기존 솔루션이 왜 그렇게 설계되었는지를 분석하고, 그 한계에서 도출된 우리 설계의 근거를 정리합니다. 단순 기능 나열이 아닌 Why → What → How 번역이 핵심입니다. 하이웍스·네이버웍스 실사용 후기 및 개인 의견을 포함하여 총 13개 항목을 도출했습니다.

| 기능 항목 (대상) | 기존 설계 의도 (Why) | 기존 한계 (Symptom) | 우리 설계 (What) + 근거 |
| :--- | :--- | :--- | :--- |
| **메시지 업무 연동**<br>(하이웍스·네이버웍스) | 전통적 결재 라인과 공식 기록 보존을 위해 메신저와 업무 관리 도구를 엄격히 분리해서 설계했습니다. | 채팅 소통 후 실제 업무(결재/Task)로 전환 시 맥락이 단절되고 데이터 입력을 중복해야 합니다. | **채팅 메시지 즉시 Task 변환 + 작업 목록 사이드바 상주**<br>→ Context Switching 비용을 0으로 만드는 것이 생산성의 핵심입니다. |
| **구글 캘린더 연동**<br>(하이웍스·네이버웍스) | 자체 캘린더 생태계 유지 및 데이터 종속성 확보를 위해 외부 캘린더와의 완전 동기화를 지원하지 않습니다. | 소규모 기업 실무에서 구글 캘린더를 병행 사용하는 경우가 많으나, 양방향 동기화 기능이 부재하여 이중 관리가 발생합니다. | **구글 캘린더 완전 양방향 동기화**<br>→ 도구 전환 없이 하나의 일정 관리 환경을 유지하는 것이 소규모 팀의 핵심 니즈입니다. |
| **외부 사용자 초대**<br>(슬랙·하이웍스) | 보안 및 워크스페이스 무결성 유지를 위해 가입 및 앱 설치를 강제하는 폐쇄적 멤버십 구조를 채택했습니다. | 단기 프로젝트 협력사나 클라이언트에게 무거운 앱 설치를 요구하는 것이 심리적·기술적 장벽이 됩니다. 외부 사용자 그룹화도 불가능합니다. | **JWT 기반 매직 링크 초대 (앱 설치 불필요) + 외부 Guest 그룹화**<br>→ 설치 없이 브라우저에서 즉시 협업을 시작합니다. |
| **화면 구성 (탭 증식)**<br>(전통 그룹웨어 전반) | 멀티태스킹 지원을 위해 각 기능을 브라우저의 새로운 탭이나 창으로 열리도록 설계했습니다. | 기능을 클릭할 때마다 탭이 늘어나 집중력이 분산되고, 저사양 PC 환경에서 메모리 점유율이 급증합니다. 크로스 브라우징 호환성 문제도 발생합니다. | **SPA 기반 Split-View (분할 화면)**<br>→ 기능 이동 시에도 메인 채팅 맥락을 유지합니다. 한 화면에서 모든 현황을 파악합니다. |
| **대화 히스토리 검색**<br>(슬랙·하이웍스) | DB 부하를 줄이고 특정 시점의 메시지만 조회하도록 리스트 형태의 검색 결과를 제공합니다. | 검색 결과 클릭 시 화면이 전환되어 전후 맥락 파악을 위해 무한 스크롤을 반복해야 합니다. | **카톡식 키워드 검색 + 인라인 컨텍스트 패널 (우측 분할)**<br>→ 현재 창을 유지한 채 우측 패널에 전후 대화를 즉시 표시합니다. |
| **주소록 검색 품질**<br>(네이버웍스) | 이름·부서 중심의 기본 검색으로 내부 조직원 탐색에 최적화하여 설계했습니다. | 전화번호, 회사 카테고리 등 세분화된 검색이 불가능하고, 매칭 키워드 하이라이팅이 표시되지 않습니다. | **전화번호·카테고리 필터 검색 + 매칭 키워드 하이라이팅**<br>→ 외부 협력사가 많은 소규모 팀에서 빠른 연락처 탐색은 핵심 생산성 요소입니다. |
| **조직도 채팅 연동**<br>(하이웍스·네이버웍스) | 조직도는 '보는' 용도로 설계되어 있고, 채팅 개설은 별도 메뉴를 통해 이루어지도록 분리되어 있습니다. | 조직도 화면에서 바로 그룹 채팅방을 만드는 직관적인 경로가 없어 불필요한 단계가 추가됩니다. | **조직도·사용자 선택 시 원클릭 그룹 채팅방 생성**<br>→ 조직도 확인 → 바로 소통으로 이어지는 자연스러운 UX입니다. |
| **관리자 프라이버시**<br>(네이버웍스) | 조직 내 컴플라이언스·감사(Audit) 목적으로 관리자의 메시지 열람 권한을 허용합니다. | 관리자 계정으로 타인의 대화 내용을 열람 가능하여 직원 입장에서 심각한 프라이버시 이슈가 발생합니다. | **채팅 열람 권한 분리 + 법적 요건 시에만 별도 감사 프로세스**<br>→ 직원의 심리적 안전감을 확보합니다. |
| **파일 전송 방식**<br>(하이웍스·네이버웍스) | 파일을 자체 서버에 저장하고 미리보기 또는 압축 버전만 제공하여 트래픽·스토리지를 절감합니다. | 수신자가 원본이 아닌 압축·변환된 파일을 받게 되어 품질 손실 및 호환성 문제가 발생합니다. | **원본 파일 전송 옵션 제공**<br>→ 디자이너, 개발자 등 파일 품질이 중요한 직군 협업 시 필수 기능입니다. |
| **커스텀 자유도**<br>(하이웍스·네이버웍스) | 표준화된 UI로 학습 비용을 낮추고 지원 부담을 줄이기 위해 커스터마이징을 제한합니다. | 기업마다 다른 업무 방식과 브랜딩을 반영할 수 없어, 업무가 도구에 맞춰지는 역전 현상이 발생합니다. | **알림·레이아웃·테마 커스터마이징 지원**<br>→ 소규모 기업이 자신의 방식대로 도구를 조정할 수 있어야 실제 채택률이 높아집니다. |
| **이메일 호환성**<br>(하이웍스) | 자체 메일 서버 기준으로 RFC 표준을 구현하였으나, 루트 도메인의 CNAME·MX 레코드 공존을 허용하는 구조로 운영합니다. | 루트 도메인에 CNAME 레코드가 존재할 경우 공공기관 메일 서버가 RFC 표준 위반으로 간주하여 메일 수신이 차단됩니다. | **도메인 연동 헬스체크: CNAME·MX 충돌 사전 감지 및 경고**<br>→ 관리자 페이지에서 선제적으로 감지하고 안내합니다. |
| **설문조사 재참여**<br>(개인 의견) | 설문 무결성 유지를 위해 한 번 제출 후 수정을 허용하지 않는 단방향 구조로 설계했습니다. | 응답자가 실수로 제출한 경우 수정할 방법이 없어 불필요한 재설문이 생성되거나 데이터가 오염됩니다. | **설문 완료 후 다시하기(수정) 기능 제공**<br>→ 소규모 팀 내부 설문에서는 무결성보다 편의성이 우선합니다. 관리자 설정으로 허용 여부를 제어합니다. |
| **발송자 클릭 액션**<br>(개인 의견) | 발송자 프로필은 정보 표시 목적으로만 설계되어 단순 이름·사진·부서 정보만 제공합니다. | 발송자 이름을 클릭해도 프로필 조회 외 추가 액션으로 이어지는 경로가 없어 별도 메뉴를 찾아야 합니다. | **발송자 클릭 시 컨텍스트 메뉴 제공**<br>→ 메일 쓰기·일정 보기·1:1 채팅·조직도 보기 등 최소 클릭으로 완성하는 UX입니다. |
| **커스텀 도메인 검증 (방어적 설계)**<br>(하이웍스·네이버웍스) | 사용자의 자유로운 DNS 설정을 허용하고, 시스템은 수동적으로 MX 레코드의 존재 여부만 체크함. | 사용자가 루트 도메인에 CNAME을 설정하는 아키텍처적 오류를 범해도 시스템이 경고하지 않아, 공공기관 메일 수신이 영구 차단되는 치명적 장애가 발생함. | **도메인 헬스체크 API 연동 (CNAME 충돌 감지)**<br>→ 비전문가인 중소기업 관리자가 인프라 설정 오류로 비즈니스 손실을 겪지 않도록, 시스템이 선제적으로 오류를 감지하고 차단하는 '방어적 설계' 적용. |

## 산출물 B-1 — 페르소나 카드

타겟은 **중소기업 (40인 이하) 및 1인·소규모 기업**입니다. 대기업용 그룹웨어의 과한 기능과 높은 비용에서 소외된 사용자군을 공략합니다. 페르소나는 Primary 2인 + Supporting 1인으로 구성했습니다.

### 🧑‍💼 이민수 (35세) — PRIMARY · 중소기업 IT 기획팀장

| 항목 | 내용 |
| :--- | :--- |
| **프로필** | 직원 40명 규모 에이전시 팀장. 5~7개 프로젝트를 동시 관리하며 메신저 소통이 하루 업무의 70%를 차지합니다. |
| **목표** | 채팅방에서 쏟아지는 수많은 업무 요청과 피드백을 누락 없이 Task로 만들고 진척도를 한눈에 보고 싶어 합니다. |
| **고충** | 업무 지시가 메신저 스레드에 파묻혀 휘발됩니다. Jira 등 외부 툴에 따로 옮겨 적는 과정이 번거롭고, 팀원들도 툴 전환을 귀찮아하여 업데이트를 누락합니다. |
| **기술력** | 상 — 다양한 SaaS 협업 툴 사용 경험이 풍부합니다. |
| **제약** | 새로운 무거운 협업 툴 도입 시 팀원들의 학습 저항감과 라이선스 비용 제약이 큽니다. |

> **💡 설계 시사점:** 메신저 중심 가벼운 업무 관리를 제안합니다. 채팅 메시지 우클릭 시 Task 즉시 할당을 지원합니다. 채팅창 우측 사이드바에 작업 목록 패널을 동시 배치합니다 (SPA 구조).

### 👩‍💻 최유진 (29세) — SECONDARY · 1인 기업 (프리랜서 마케터)

| 항목 | 내용 |
| :--- | :--- |
| **프로필** | 개인 사업자. 3~4개의 클라이언트 기업 담당자 및 여러 외부 프리랜서(디자이너, 개발자 등)와 단발성 프로젝트를 진행합니다. |
| **목표** | 여러 외부 협력사와의 커뮤니케이션을 카톡(개인용)과 분리하여, 하나의 업무용 툴에서 깔끔하게 그룹화해 관리하고 싶어 합니다. |
| **고충** | 대기업용 그룹웨어는 기능이 너무 많고 비쌉니다. 외부 사용자를 초대할 때마다 복잡한 권한 설정이나 앱 설치를 강요하기 미안해합니다. |
| **기술력** | 중 — 카카오톡, 노션 수준의 대중적 툴에 익숙합니다. |
| **제약** | 도입 및 유지 비용이 매우 저렴해야 하며, 외부인이 별도 교육 없이 즉시 쓸 수 있을 만큼 직관적이어야 합니다. |

> **💡 설계 시사점:** 주소록 기반 외부 Guest 초대·그룹화를 적용합니다. 이메일 링크 클릭만으로 참여 가능하도록 설계합니다 (JWT 매직링크). 카톡과 유사한 직관적 UI를 구성합니다.

### 👦 박지훈 (26세) — SUPPORTING · 중소기업 신입사원 (입사 2개월)

| 항목 | 내용 |
| :--- | :--- |
| **프로필** | 입사 2개월 차 제조/유통 중소기업 신입. 업무 파악과 사내 인맥 형성이 최우선 과제입니다. |

## 산출물 B-2 — 고객 여정 지도 (CJM)

가이드라인 요구사항인 4가지 축 (단계 / 접점 / 감정 곡선 / 보조 지표)을 반영합니다. 감정 곡선이 급락하는 Pain Point가 곧 우리 설계의 핵심 과제입니다. 메신저 기능 중심으로 작성했습니다.

### 1️⃣ [이민수 · PRIMARY] 채팅 지시사항의 업무화 및 추적 여정

| 축 | 내용 |
| :--- | :--- |
| **단계** | 업무 수신 (채팅) → 내용 파악 및 분리 → 업무 할당 (타 툴) → 진척도 확인 → 완료 보고 |
| **접점** | 메신저 푸시 알림 → 긴 스레드 읽기 → 별도 앱(Jira/Excel) 입력 → 메신저로 진행상황 묻기 → 완료 답변 |
| **감정** | 보통 → **최저(짜증/막막)** → 귀찮음 → 피곤함 → 안도 |
| **지표** | 채팅 내용 파악 소요 시간 / Context Switching 횟수 / 지시사항 누락 건수 |

* **🎯 Pain Point:** [내용 파악 → 업무 할당] 구간. 스레드가 길어지면 정보의 블랙홀이 되고, 중요한 지시를 별도 툴에 옮겨 적는 과정에서 컨텍스트 스위칭이 발생하여 업무 누락 위험이 큽니다.
* **🚀 해결 전략:** 채팅 메시지의 즉각적인 Task 변환 기능을 추가합니다. 메시지 클릭 한 번으로 담당자를 지정하고 업무를 할당합니다. 채팅방 우측 '작업 목록 사이드바'에 실시간으로 연동합니다 (SPA 아키텍처).

### 2️⃣ [박지훈 · SUPPORTING] 과거 업무 히스토리 검색 여정

| 축 | 내용 |
| :--- | :--- |
| **단계** | 키워드 검색 → 결과 목록 확인 → 특정 메시지 클릭 → 전후 맥락 파악 → 업무 적용 |
| **접점** | 메신저 상단 검색창 → 검색 결과 리스트 → **새 탭으로 열리는 원본 채팅방** → 위아래 무한 스크롤 → 기존 화면 복귀 |
| **감정** | 보통 → 기대 → **최저(혼란/짜증)** → 피곤함 → 보통 |
| **지표** | 검색 후 새 탭 생성 개수 / 원하는 맥락 찾기까지의 스크롤 횟수 / 검색 포기율 |

* **🎯 Pain Point:** [특정 메시지 클릭 → 전후 맥락 파악] 구간. 기존 솔루션은 검색 결과를 누르면 새 탭이 열리거나 스레드 뷰로 튕겨버려, 원래 하던 작업의 맥락이 완전히 끊어집니다.
* **🚀 해결 전략:** 새 탭 없는 Split View 통합 검색을 개발합니다. 검색 결과 클릭 시 화면 이동 없이 우측 슬라이드 패널에 해당 메시지의 전후 맥락(History)을 렌더링합니다.

### 3️⃣ [최유진 · SECONDARY] 외부 협력사 메신저 초대 여정

| 축 | 내용 |
| :--- | :--- |
| **단계** | 프로젝트 방 생성 → 외부 인원 초대 발송 → 상대방의 참여 시도 → 권한/프로필 세팅 → 소통 시작 |
| **접점** | 그룹웨어 주소록/방 만들기 메뉴 → 이메일/카톡 초대 링크 → **(상대방 시점) 회원가입 및 전용 PC앱 설치 강요 화면** → 관리자 권한 승인 → 방 진입 |
| **감정** | 기대 → 보통 → **최저(미안함/난감)** → 피곤함 → 안도 |
| **지표** | 초대 → 최초 메시지 송수신까지 소요 시간 / 외부 게스트 가입·설치 이탈률 / 권한 설정 관련 문의 횟수 |

* **🎯 Pain Point:** [외부 인원 초대 발송 → 상대방의 참여 시도] 구간. 1~2달짜리 단발성 프로젝트를 위해 클라이언트에게 무거운 전용 그룹웨어 앱 설치나 복잡한 회원가입을 요구하는 것이 비즈니스 상 매우 껄끄러운 경험으로 남습니다.
* **🚀 해결 전략:** 앱 설치 불필요 '링크 기반 즉시 참여 (Guest Access)' 기능을 구축합니다. 이메일만 입력하면 만료 시간·해당 채팅방 접근 권한만 담긴 JWT 매직 링크를 발송합니다. 브라우저(SPA)로 회원가입 없이 즉시 대화에 참여하게 만듭니다.

## 산출물 C — 핵심 유저 스토리 (요구사항 명세)

### 📊 요구사항 우선순위 매트릭스 요약

| 분류 | Quick Wins (높은 가치, 적은 노력) | Flagship (높은 가치, 많은 노력) | Fill-ins (낮은 가치, 적은 노력) |
| :--- | :--- | :--- | :--- |
| **Must** (필수) | 로그인, 직급 관리, 부서 관리, 채팅방 생성, 원본 파일 전송 | 실시간 메시지 전송, Task 변환, 일정 변환, 게스트 매직링크 | - |
| **Should** (권장) | 발송자 퀵 액션 메뉴, 비밀번호 초기화 | Split-View 통합 검색, 구글 캘린더 양방향 동기화 | 프로필 상태 표시 |
| **Could** (선택) | - | - | 알림 커스텀, 설문조사 수정 |

### 📝 유저 스토리 상세 명세 (Acceptance Criteria 포함)

#### [US-001] 이메일 기반 로그인 (Quick Wins)
* **As a** 사용자
* **I want to** 이메일과 비밀번호로 시스템에 로그인할 수 있다.
* **So that** 나의 권한에 맞는 그룹웨어 기능에 접근할 수 있다.
* **우선순위:** Must 
* **인수 조건:**
  * 이메일 형식이 아닐 경우 "이메일 형식을 확인하세요" 경고가 실시간(Focus out 시) 표시된다.
  * 5회 연속 비밀번호 불일치 시 계정이 30분간 잠금 처리되며, DB의 `locked_until` 타임스탬프가 갱신된다.
  * 로그인 성공 시 클라이언트에 Access Token(30분)과 Refresh Token(7일)이 JWT 형태로 발급된다.

#### [US-002] 비밀번호 초기화 (Quick Wins)
* **As a** 사용자
* **I want to** 로그인 화면에서 비밀번호 찾기를 요청할 수 있다.
* **So that** 비밀번호를 잊어버렸을 때 스스로 복구할 수 있다.
* **우선순위:** Should
* **인수 조건:**
  * 가입된 이메일 입력 시 15분간 유효한 1회성 초기화 링크가 발송된다.
  * 이전과 동일한 비밀번호로 변경을 시도하면 "기존과 다른 비밀번호를 입력해주세요" 알림을 노출한다.

#### [US-003] 부서 조직 관리 (Quick Wins)
* **As a** 관리자
* **I want to** 사내 조직도에 새로운 부서를 추가, 수정, 삭제할 수 있다.
* **So that** 회사의 최신 조직 구조를 시스템에 반영할 수 있다.
* **우선순위:** Must
* **인수 조건:**
  * 부서 추가 시 트리(Tree) 구조의 최하단에 즉시 렌더링되어야 한다.
  * 하위 부서나 소속 직원이 있는 부서는 삭제할 수 없으며, "소속된 직원이나 하위 부서를 먼저 이동시켜주세요" 팝업을 띄운다.

#### [US-004] 직급 추가 및 삭제 (Quick Wins)
* **As a** 관리자
* **I want to** 사내 직급 체계를 추가하거나 삭제할 수 있다.
* **So that** 유연한 인사 제도를 시스템 권한에 연동할 수 있다.
* **우선순위:** Must
* **인수 조건:**
  * 중복된 직급명 입력 시 등록 버튼이 활성화되지 않는다.
  * 해당 직급에 매핑된 사용자가 0명일 때만 삭제가 가능하다. 1명 이상일 경우 에러 코드(409 Conflict)를 반환한다.

#### [US-005] 조직도 기반 채팅방 생성 (Quick Wins)
* **As a** 직원
* **I want to** 조직도에서 특정 부서 전체를 체크하여 채팅방을 생성할 수 있다.
* **So that** 팀원들을 일일이 검색해서 초대하는 수고를 덜 수 있다.
* **우선순위:** Must
* **인수 조건:**
  * 상위 부서 체크박스 클릭 시 하위 소속원이 모두 자동 선택된다.
  * 본인을 제외한 초대 인원이 0명일 경우 "초대할 인원을 선택하세요" 모달을 띄우고 채팅방 생성을 차단한다.

#### [US-006] 웹소켓 기반 실시간 메시지 전송 (Flagship)
* **As a** 직원
* **I want to** 채팅방에서 텍스트 메시지를 전송하고 즉시 수신할 수 있다.
* **So만 that** 지연 없는 실시간 업무 소통이 가능하다.
* **우선순위:** Must
* **인수 조건:**
  * 메시지 전송 시 화면 새로고침 없이 채팅창 하단에 즉시 말풍선이 렌더링된다.
  * 네트워크 끊김 시 전송 대기(Loading) 아이콘이 표시되며, 복구 시 자동 재전송된다.

#### [US-007] 원본 파일 전송 및 미리보기 (Quick Wins)
* **As a** 디자이너/개발자
* **I want to** 채팅창에 파일을 압축 없이 원본 화질로 전송할 수 있다.
* **So that** 작업물의 품질 손실 없이 팀원들과 피드백을 주고받을 수 있다.
* **우선순위:** Must
* **인수 조건:**
  * 이미지(jpg, png) 첨부 시 채팅창 내에 압축되지 않은 해상도로 미리보기가 생성된다.
  * 단일 파일 용량이 50MB를 초과할 경우 업로드를 차단하고 경고 툴팁을 표시한다.

#### [US-008] 채팅 메시지의 Task(작업) 변환 (Flagship)
* **As a** 프로젝트 매니저
* **I want to** 특정 채팅 메시지를 클릭하여 '작업 할당' 버튼을 누를 수 있다.
* **So that** 메신저 내에서 지시사항을 즉시 작업 목록으로 이관하여 추적할 수 있다.
* **우선순위:** Must
* **인수 조건:**
  * 메시지 옵션 `...` 클릭 시 `작업 할당` 메뉴가 표시된다.
  * 할당 완료 시 채팅창 우측 사이드바(작업 패널)에 즉시 동기화되며 상태는 '대기'로 자동 설정된다.
  * 원본 채팅 메시지 하단에 `[작업으로 등록됨]` 태그가 뱃지 형태로 부착된다.

#### [US-009] 할 일 목록 필터링 옵션 (Side-panel)
# * As a 프로젝트 매니저
# * I want to 사이드바의 할 일 목록에서 '현재 채팅방 업무'와 '전체 업무'를 전환하여 볼 수 있다.
# * So that 업무 범위를 상황에 맞게 빠르게 전환하며 관리할 수 있다.
# * 우선순위: Should
# * 인수 조건:
#   - 사이드바 상단에 `필터 전환` 토글 버튼이 존재해야 한다.
#   - `전체 보기` 모드에서는 내가 담당자로 지정된 모든 채팅방의 할 일이 리스트업된다.
#   - 각 할 일 항목 클릭 시, 해당 업무가 생성된 원본 채팅방의 특정 시점(Context)으로 즉시 이동(Jump)해야 한다.

#### [US-010] 외부 게스트 매직 링크 초대 (Flagship)
* **As a** 1인 기업
* **I want to** 이메일 주소만으로 외부 클라이언트를 내 채팅방에 초대할 수 있다.
* **So that** 상대방에게 앱 설치나 가입을 강요하지 않고 협업을 시작할 수 있다.
* **우선순위:** Must
* **인수 조건:**
  * 게스트는 이메일로 받은 링크 클릭 시 브라우저에서 바로 채팅방에 입장한다.
  * 게스트용 JWT 토큰은 해당 채팅방(Room ID) 이외의 사내 조직도, 타 채팅방에 접근 시 403 Forbidden 에러를 반환한다.

#### [US-011] 발송자 프로필 퀵 액션 (Quick Wins)
* **As a** 직원
* **I want to** 채팅방에서 상대방의 프로필 사진을 클릭하여 바로 메일이나 전화를 걸 수 있다.
* **So that** 연락처를 따로 검색하는 단계를 생략할 수 있다.
* **우선순위:** Should
* **인수 조건:**
  * 프로필 클릭 시 '1:1 채팅', '이메일 쓰기', '일정 보기' 컨텍스트 메뉴가 팝업된다.
  * 이메일 쓰기 클릭 시 수신자 란에 해당 직원의 이메일이 자동 기입된 폼이 열린다.

#### [US-012] Split-View 통합 검색 (Flagship)
* **As a** 신입사원
* **I want to** 검색 결과 클릭 시 새 탭이 열리는 대신 우측 패널에서 전후 대화를 볼 수 있다.
* **So that** 현재 하던 채팅의 맥락을 잃지 않고 과거 기록을 참고할 수 있다.
* **우선순위:** Should
* **인수 조건:**
  * 검색어와 일치하는 키워드는 노란색 하이라이트 CSS가 적용된다.
  * 결과 클릭 시 우측 슬라이드 패널이 열리며, 원본 메시지 기준 위아래 10개의 히스토리를 불러온다.

#### [US-013] 구글 캘린더 양방향 동기화 (Flagship)
* **As a** 소규모 기업 직원
* **I want to** 그룹웨어 캘린더와 개인 구글 캘린더를 연동할 수 있다.
* **So that** 이중으로 일정을 등록하는 수고를 덜 수 있다.
* **우선순위:** Should
* **인수 조건:**
  * OAuth 2.0을 통해 구글 계정 연동 시, 그룹웨어에 등록한 일정이 구글 캘린더에 즉시 반영된다 (Create, Update, Delete 지원).

#### [US-014] 상태 표시 (Fill-ins)
* **As a** 직원
* **I want to** 팀원들의 프로필에 현재 상태(자리비움, 회의중 등)를 볼 수 있다.
* **So that** 채팅을 보내기 적절한 타이밍인지 알 수 있다.
* **우선순위:** Should
* **인수 조건:**
  * 상태 변경 시 웹소켓을 통해 조직도 및 채팅방 내 모든 해당 유저 프로필 아이콘 우측 하단에 상태 인디케이터(색상)가 실시간 업데이트된다.

#### [US-015] 채팅방별 알림 커스터마이징 (Fill-ins)
* **As a** 직원
* **I want to** 중요하지 않은 채팅방의 알림만 끌 수 있다.
* **So that** 잦은 알림으로 인한 업무 방해를 줄일 수 있다.
* **우선순위:** Could
* **인수 조건:**
  * 알림 끄기 설정 시, 해당 방에서 새 메시지가 와도 브라우저 푸시 알림 및 소리가 발생하지 않는다 (안읽음 뱃지 숫자만 증가).

#### [US-016] 설문조사 다시하기 (Fill-ins)
* **As a** 직원
* **I want to** 제출한 설문조사의 응답 내용을 마감 전까지 수정할 수 있다.
* **So that** 잘못 선택한 응답을 바로잡아 데이터를 정확하게 유지할 수 있다.
* **우선순위:** Could
* **인수 조건:**
  * 설문 작성자가 '수정 허용' 옵션을 켠 설문에 한해, 제출 후 `응답 수정` 버튼이 노출된다.
  * 수정 후 재제출 시 DB에 신규 Row가 삽입되는 것이 아니라 기존 응답 레코드가 Update 되어야 한다.
